# Contributing

Thank you for helping improve this product.

## Workflow

1. Open an issue or discussion before large changes.
2. Create a focused branch from `main`.
3. Keep commits small and descriptive.
4. Run the repository's verification commands before opening a pull request.
5. Open a PR with a clear summary, test evidence, and screenshots/logs when relevant.

## Commit Convention

Use concise conventional commits when possible:

- `feat: add ...`
- `fix: resolve ...`
- `docs: update ...`
- `test: cover ...`
- `chore: maintain ...`

## Pull Request Checklist

- [ ] Scope is limited and described clearly.
- [ ] README/CHANGELOG/docs are updated when behavior changes.
- [ ] Security-sensitive changes are called out explicitly.
- [ ] Local tests or CI checks pass.
- [ ] No secrets, credentials, private URLs, or internal-only identifiers are committed.

## Governance

Repository governance follows the AtomCollide / 智械工坊 GitHub repository SOP. High-risk operations such as branch deletion, tag deletion, permission changes, repository archival, and branch protection changes require an explicit confirmation manifest before execution.
